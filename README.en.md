## Project Details

### Live Caption Teleprompter

`live-teleprompter/` is a browser-based teleprompter for spoken recordings, live streaming, presentations, and scripted delivery. Users can paste or save a script, read from it, and let the app match the spoken text to the current script position so the page scrolls along with the speaker.

Key capabilities:

- **Voice-follow scrolling**: Uses browser speech recognition to listen to the speaker, match the transcript against the script, and scroll to the current segment.
- **Fixed-speed scrolling**: Provides a speed-based scrolling mode when speech recognition is unavailable or not needed.
- **Script management**: Supports saving, opening, and deleting scripts locally for repeated recording workflows.
- **Reading controls**: Includes font size, matching sensitivity, lead characters, previous/next segment navigation, manual sync, and reset controls.
- **Fullscreen mode**: Offers a focused reading interface for recording and presentation scenarios.
- **Diagnostics**: Includes microphone permission checks, input volume testing, simulated transcript input, match confidence, and event logs for troubleshooting speech recognition.

Runtime notes:

- The app is a pure frontend static page. The published directory contains `index.html`, the main application bundle, styles, and a matching-algorithm worker.
- Voice-follow mode depends on the browser's `SpeechRecognition` / `webkitSpeechRecognition` support, which varies by browser.
- Microphone access and speech recognition usually require HTTPS or another secure context; the GitHub Pages URL satisfies this requirement.
- Scripts and user preferences are stored in browser local storage and do not require a backend service.

## Publishing

This repository is intended for static hosting only. It does not contain application source code, dependency installation steps, or local build scripts. The recommended workflow is:

1. Build the application in its source repository.
2. Copy the generated build output into the target directory in this repository.
3. Confirm that the entry file and asset references use relative paths so they work under a GitHub Pages subpath.
4. Commit and push the changes to GitHub, where GitHub Pages serves the files.

## Maintenance Guidelines

- Keep each published target in its own directory to avoid asset collisions.
- Preserve hashed filenames generated by the build tool for cache invalidation and version tracking.
- When updating assets, verify that every file referenced by `index.html` exists.
- When adding a new static page, update the "Currently published" table with its path, description, and URL.
- Do not store source-level dependencies, temporary files, secrets, or local environment configuration in this repository.

## Local Preview

To quickly inspect the static files, start any static file server from the repository root:

```bash
python3 -m http.server 8000
```

Then open:

```text
http://localhost:8000/live-teleprompter/
```
